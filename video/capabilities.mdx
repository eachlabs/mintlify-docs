<Note>
**Private beta.** This page is shared by direct link only. The surface below is validated server-side with `additionalProperties: false` ŌĆö a request is either exactly this shape or it is rejected before any compute runs, at $0. Contracts here are stable for the beta; additions may land, removals will not without notice.
</Note>

Every capability job is submitted the same way: `POST https://api.eachlabs.ai/v1/prediction` with header `X-API-Key`, body `{"model": "eachlabs-video-api", "input": {"capability": "<name>", "tier": "cpu", "input_url": "...", "params": {...}}}`, then poll `GET /v1/prediction/{id}` until a terminal status. `tier` is **required** (`cpu` or `gpu`). Analysis capabilities (`probe`, `scene_detect`, `silence_detect`) return structured JSON in `output`; everything else returns hosted artifact URL(s). See [Quickstart](/video/quickstart) for the input URL requirements and [Billing & Limits](/video/billing-limits) for pricing mechanics. For long-tail cases the named capabilities don't cover, a raw-ffmpeg mode with a different top-level contract is documented at [run_ffmpeg](#run_ffmpeg), and multi-step patterns are collected under [Recipes](#recipes).

<Warning>
`tier: "gpu"` is a **beta tier** ŌĆö capacity is limited and several codecs are cpu-only (rejected on gpu, noted per capability). Use it where a capability supports it; fall back to `cpu` for anything gpu rejects.
</Warning>

## audio_replace

Swap or mix a video's audio track using a second input as the audio source; video always stream-copied.

**Inputs:** `input_urls` (2) ┬Ę **Tiers:** `cpu` ┬Ę `gpu` ┬Ę **Output:** media ŌĆö single file

<Note>**GPU behavior:** tier-agnostic (video copied, aac everywhere)</Note>

| Parameter | Type | Constraints | Default | Required | Notes |
|-|-|-|-|-|-|
| `mode` | string | `replace` \| `mix` | `"replace"` | no | replace: -shortest (output runs to the shorter stream); mix: amix ending with the shorter audio over the full video. A base video with **no audio stream** is accepted for `replace` ŌĆö the new track is simply added |
| `container` | string | `mp4` \| `mkv` | `"mp4"` | no |  |
| `normalize` | boolean |  | `true` | no | mix only: true halves each track (no-clip-safe); false preserves source levels |

```json
{
  "model": "eachlabs-video-api",
  "input": {
    "capability": "audio_replace",
    "tier": "cpu",
    "input_urls": [
      "<YOUR_INPUT_URL_1>",
      "<YOUR_INPUT_URL_2>"
    ],
    "params": {
      "mode": "replace",
      "container": "mp4",
      "normalize": true
    }
  }
}
```

<Note>
There is no start-offset parameter. To delay the incoming audio, pre-pad it locally before upload: `ffmpeg -i narration.mp3 -af "adelay=3000|3000" padded.mp3` adds 3s of leading silence. MP3 padding carries ~46ms of encoder-delay skew ŌĆö pad as WAV or AAC when you need frame-accurate sync.
</Note>

## run_ffmpeg

Raw ffmpeg/ffprobe for long-tail cases the 37 named capabilities don't cover. **Available to beta organizations today**; access may move to per-key grants later in the beta.

**Different contract.** `run_ffmpeg` is a top-level `mode`, not a capability ŌĆö the input carries `mode`, `argv`, `input_url`, `output_format`, and `tier` (no `capability`, no `params`):

```json
{
  "model": "eachlabs-video-api",
  "input": {
    "mode": "run_ffmpeg",
    "tier": "cpu",
    "input_url": "<YOUR_INPUT_URL>",
    "output_format": "mp3",
    "argv": ["ffmpeg", "-hide_banner", "-nostats", "-i", "{input}", "-af", "acompressor=threshold=-18dB:ratio=2:attack=30,loudnorm=I=-14:TP=-1.0:LRA=11,alimiter=limit=0.891:level=false", "-ar", "44100", "-c:a", "libmp3lame", "-b:a", "320k", "{output}"]
  }
}
```

The example above is a working audio-mastering command (compress ŌåÆ normalize to -14 LUFS ŌåÆ true-peak limit ŌåÆ 320k MP3). Two gotchas it encodes:

- `alimiter` needs `level=false` ŌĆö the default (`level=true`) re-normalizes after limiting and **silently defeats the true-peak ceiling**.
- The explicit `-ar 44100` is required ŌĆö `loudnorm` internally upsamples to 192kHz, and without an explicit rate the output stays at 192kHz.

**Placeholders.** Never write media paths or URLs into `argv` ŌĆö staged placeholder paths are the only way to reach your media, and the only literal path a job needs is the `/dev/null` sink shown below. The server substitutes `{input}` (the staged input), `{input0}`, `{input1}`, ŌĆ” (indexed inputs), and `{output}` (the artifact path, whose extension comes from `output_format`).

**The enforced safety boundary.** Commands are validated server-side, and ffmpeg runs with an injected `-protocol_whitelist file` ŌĆö commands can read local files only, never the network. Every `run_ffmpeg` job is billed like any capability job. The argv restrictions:

- `argv[0]` must be `ffmpeg` or `ffprobe`.
- Denied tokens include: URL schemes (`http://`, `https://`, ŌĆ”), `movie=` / `amovie=` filter sources, `-protocol_whitelist` overrides, and bare `-` (stdin/stdout). Placeholder-bound paths reject `..` traversal.

**ffprobe.** With `argv[0]: "ffprobe"` the JSON is returned inline in the prediction `output` ŌĆö no media artifact.

**Analysis-to-file pattern.** Filters that print metadata can write their report to the artifact instead of producing media ŌĆö point the filter's `file=` at `{output}` and satisfy ffmpeg's output requirement with the `-f null /dev/null` sink:

```json
"output_format": "vtt",
"argv": ["ffmpeg", "-i", "{input}", "-af", "ametadata=mode=print:file={output}", "-f", "null", "/dev/null"]
```

<Note>
**Two-pass loudnorm.** Single-pass `loudnorm` is dynamic and can land off-target. For mastering-grade accuracy run a first pass with `loudnorm=...:print_format=json` through the `-f null /dev/null` sink to measure, then a second pass feeding back `measured_I`, `measured_LRA`, `measured_TP` with `linear=true` ŌĆö this lands within 0.5 LU of target.
</Note>

## Recipes

Multi-step patterns verified end-to-end against production.

### Clip list + first frames: `segment` ŌåÆ `stills`

Split with `segment`, then run `stills` with `position: "first"` on each chunk ŌĆö segment output URLs are directly usable as the next job's `input_url`. For boundary-accurate frames (to within one frame) set `reencode: true` on the segment job: stream-copy segmentation cuts only on keyframes, and measured chunk durations drifted to 33.36s / 33.33s / 23.33s against a requested 30/30/30 split.

### Split on silence with a max-duration cap: `silence_detect` ŌåÆ `trim` per segment

1. Run `silence_detect` on the source.
2. Compute capped cut points client-side: cut at each silence's `end`; hard-cut when a segment would exceed your max duration; skip segments shorter than your min; cap the tail.
3. Submit one `trim` per computed segment.

<Note>
Stream-copy `trim` carries +0.00ŌĆō0.17s of end drift per segment. When exact boundaries matter, use `silence_remove` (keep-ranges, re-encodes) instead of per-segment trims.
</Note>
